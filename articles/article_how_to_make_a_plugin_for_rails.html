How To Publish A Rails Plugin, In Five Easy Steps

by Kevin J. Storberg

Published on: January 2, 2019

Rails is a pretty robust framework. I often find myself in awe at just how incredible it really is. But let's face it, sometimes you need something more.

Recently, I was thinking it would be terrific to log instances of my models to the console in a more readable format. I am finicky about code and content being easy to read, thus it was only natural that I extended that disposition. So I wrote a really cool module which I named "SnoopDogg". It alphabetizes the columns in my models and neatly prints them to the console, color coded by data type.

I was quite satisfied with my work and decided that it would be a terrific little snippet to share with the Rails community at large. So I turned it into a plugin which can be found here.

This had me wondering how many cool modules and features have been created by developers and never shared, simply because they did not realize how easy it really is. So I decided to write this post to explain how to generate a simple Rails plugin, hoping that it will lead to some cool new gems for the community to use.

I outline the steps to generate, and publish the gem below (excluding testing). These steps assume you already have the latest version of Rails and Ruby installed.

Step 1: Name your gem

Naming your gem is an important step because the community has naming conventions for ruby gems. Naming conventions give fellow developers a clue about what is under a gem's hood.

In general, we try to follow these pretty simple rules:
  • Use underscore for multiple words
  • Use dashes for extensions
  • Don't use uppercase letters

If you want to read more about naming conventions, you can do so here.

SnoopDogg was pretty simple. Camelcase modules and classes are converted into snake_case when naming a gem. Hence, SnoopDogg => snoop_dogg.

Be sure to double check that your gem’s name has not already been taken before moving onto step 2.

Step 2: Generate The Repository

The step is probably the easiest. Use your shiny new name on Github or some other origin to create a repo. Then, in your local terminal, run "rails plugin new gem_name". In my case, "rails plugin new snoop_dogg".

Poof! You have the scaffolding for a fancy new gem! It still has that new gem smell! Now initialize a local git repo, and push your initial commit to origin like you would do for any other project.

Step 3: Writing The Code

This is where it can get tricky, depending on the complexity of the code you intend to share. We are going to look at a fairly simple case with SnoopDogg, but the general principles can be extended for more complex cases.

In your gem's directory, you will find a file called "gem_name/lib/gem_name.rb". This is where you will place the code you intend to share. For SnoopDogg, this could have been simple enough, all I needed to do was paste my already written module. However, I wanted to go a step further and automatically include the SnoopDogg module in every model of an application downloading the gem. Here is simple way to do just that:

Just under the module, in "snoop_dogg/lib/snoop_dogg.rb", I wrote:

ActiveRecord::Base.send(:include, SnoopDogg)

Boom! Now SnoopDogg will automatically be included in every model in an application which uses it.

Step 4: Documentation

Alas, we are done writing the code and ready to write the boring, yet very important documentation. This is where you will list your dependencies, make acknowledgments, define your licensing, and provide directions to use and even contribute to your gem.

To start, we need to define the gem. In ‘gem_name/gem_name.gemspec’, you will find a short list of attributes which you can use to specify some of the most important information about your gem. Do not go crazy here, we will use the README.md to go into greater detail. Just be sure to include the following:

  • The author(s) - This takes an array of strings
  • The homepage - The repo url where the code is stored
  • A contact email - Only put an email you are comfortable making public
  • A brief description - Again do not go crazy, just something to the point
  • Any dependencies - This is important because bundler will need this
  • The licensing - SnoopDogg is using the MIT license

Now let's take advantage of the README.md file so it can be referenced as documentation. Here, you will once again reference the authors, homepage, description, dependencies, and licensing. But the README.md is where you can (and should) go into greater detail. You can write a detailed description along with directions on how to install and use your gem. You can link to the pages of your authors, or the documentation of any dependencies. You can even provide information on how to contribute.

If you do not know how to write a README.md file, you can teach yourself here. I would suggest taking the time to learn the syntax. It is quite easy to learn, and knowing it will allow your gem’s public repositories to display your README.md in a way that is pleasing to the eyes.

Finally, visit ‘gem_name/lib/gem_name/version.rb’ to update the version number of your gem to reflect your work

Step 5: Publish

Now it is time to share our shiny new gem with the world. We will want to publish the gem on rubygems.org. This is where bundler commands will look when someone attempts to install your gem. To do this, you will need an account with rubygems.org. If you do not have a rubygems.org account, make one now. I will wait here.

Now that you have an account, you will need to ‘build’ the latest version your gem. From your gem’s directory, run ‘gem build gem_name.gemspec’. In my case, I ran ‘gem build snoop_dogg.gemspec’. A file will appear in your repository matching the naming convention ‘gem_name-version.gem’. My file was named ‘snoop_dogg-0.1.3.gem’.

This file contains the information about this version of our gem. When we pass it along to rubygems.org they will pass that information along to anybody that downloads the gem. As the gem is updated and new versions are released, rubygems.org will make sure that any downloads are for the latest release, OR the version requested.

From that same directory, run ‘gem push gem_name-version.gem’. You will be prompted to enter your rubygems.org credentials so that rubygems can give you credit for your contribution. Enter your username and password. Done!

Yes, done! That's it! You are done! It is as easy as 1, 2, 3, 4, 5. (That is how that saying goes right?).
